Showing Claude Code dynamic-workflow runs in kolu

Analysis of a live do-wf run in drishti/.worktrees/hm, generalized to the real case: a session holds many workflow runs at once — some running, some finished. Two-phase proposal, revised after hickey + lowy.

Headline: kolu already knows a workflow is running — it just can't say which one or how many agents. The running state works (it comes from the transcript, not the journal). The enrichment is what's broken: deriveWorkflowProgress() reads a journal file that no longer exists, returns null, so the name/fan-out badge stays dark. And the model is single-run — even fixed, it collapses a session's whole history to one entry.
verified: core.ts:493-552 reads one dead workflows/<runId>.json; state is set separately at core.ts:378-385
Precisely what fails (this is not "kolu thinks nothing is running"). deriveState promotes waiting → running_background purely from the main-transcript launch marker via outstandingBackgroundTasks() — no journal read core.ts:378-385. So the spinner + "Running in background" label already render. The journal is read only to fill the workflow field session-watcher.ts:245-247, and agentWorkflow() needs both state===running_background and workflow!=null agentDisplay.ts:43-44. State is true; workflow is null → the do-wf · 5 agents badge in TerminalMeta.tsx:80 and the "Workflow" row in MetadataInspector.tsx:194 are the only things dark.

What a dynamic workflow writes, and where status actually lives

The subtle one — status is a transcript fact, not a journal fact. The journal records per-agent started/result, never a "workflow finished" event. I verified this matters: in the live drishti run, 5 agents have resulted while the workflow is still going (it's between phases, about to spawn the next agent). So the tempting rule "all started agents resulted ⇒ completed" reports false completions between phases. The authoritative signal is in the main transcript: a <task-notification> carrying <status>completed</status>, keyed by the launch Task ID, which is linked to the Run ID in the same launch marker. In the 2-run session: 4 completed notifications + 1 running. verified: 4×<status>completed</status> in cc-workflow.jsonl; journals contain only started/result. kolu already pairs Task↔Run IDs at core.ts:419, 478.

Proposal — two phases, each shippable alone

Phase 1 — repoint the reader (the workflow name + fan-out count appear)

Bug-fix-shaped, almost entirely in core.ts; no UI change. The agent already shows as running in background — Phase 1 only restores the enrichment the dark badge/inspector row are waiting on. They already consume ClaudeWorkflow {name,status,agents} agentDisplay.ts:42-44, TerminalMeta.tsx:78-80, MetadataInspector.tsx:194-205:

  • Point the journal path at subagents/workflows/<runId>/journal.jsonl and the watcher with it.
  • Read the event log: agents = distinct started agentIds; name from the script meta (workflows/scripts/<name>-<runId>.js).
  • Status comes from the transcript, not the journal. A run is running iff its taskId is still in outstandingBackgroundTasks(); completed/failed once its terminal <task-notification> is seen. This reuses machinery that already exists — no new "all-agents-done" heuristic. core.ts:441-487

The live badge keeps showing the currently-running run (the existing single-field summary), which is the right glanceable value for the terminal tile. Multiplicity is surfaced in Phase 2.

Wiring (shaped by the review pass below):

  • On-demand fetch, returning a list. A new RPC loadWorkflowRuns(session) enumerates subagents/workflows/wf_*/ (so finished runs appear, not just outstanding ones) and returns WorkflowRun[] — each {runId, name, status, startedEpoch, endedEpoch, agents[]}. Mirrors the existing exportTranscriptHtml/loadClaudeCodeTranscript pull pattern. router.ts:13, 204-247
  • Live metadata stays scalar. ClaudeCodeInfo still carries only the single running-run summary (plus optionally a small "N runs" count) — the full list is never on the firehose.
  • Per-agent transcript reuses the canonical seam. agent-<id>.jsonl is a normal transcript → parseClaudeCodeJsonl() → TranscriptEvent[] (already models sub-agent dispatch via subtask_start) → existing kolu-transcript-html renderer. No parallel event schema. verified: transcript.ts:115, 316-339

What the hickey + lowy pass changed

Reuse the transcript IR seam (lowy headline). Claude JSONL is already normalized to TranscriptEvent[] with sub-agent boundaries. The per-agent transcript view consumes that; only the journal (a distinct runtime format) gets its own tiny reader. transcript.ts:115, 316
Keep ClaudeWorkflow scalar; never put the run list on live metadata (hickey F1 + lowy axis-split). A list of full runs on the firehose would bloat every metadata tick and multiply the unenforced "workflow non-null only while running_background" invariant already split across a write gate and a read gate. session-watcher.ts:44, 244; agentDisplay.ts:42-44 Fetched on demand instead.
No new right-panel tab (lowy). RightPanelTabKindSchema = z.enum(["inspector","code"]) is a wire contract with persisted activeTab + session restore — adding a kind is a protocol change. The run list lives inside the existing Inspector tab. verified: surface.ts:83, 95-97
Two honest gaps. (1) The journal doesn't tag each agent with its phase — grouping is inferred from start order against meta.phases, reliable for a sequential pipeline like do-wf but not arbitrary parallel() fan-out. Fallback: a flat agent timeline (fully journal-backed) plus the declared phase strip; the exact fix is to have the runtime stamp phase into the journal (upstream). (2) Status for an old completed run needs the terminal notification, which may have scrolled out of the transcript tail — so the status scan must read the full transcript (or treat dir-present + not-outstanding as completed, accepting it can't distinguish completed/failed/killed without the notification).

Cut line

Phase 1 alone is worth shipping: a small correction that makes an already-built surface work for the first time, showing the active run. Phase 2 adds the multi-run list + timelines and is independently reviewable. When ready, drive it with /do.
